The PineRE Ethos

PineRE exists to create software that is understandable, maintainable and enjoyable to build.

Every decision should improve not only the operating system itself, but also the experience of learning, maintaining and contributing to it.

Simplicity

When presented with multiple solutions, PineRE prefers the simpler architecture.

Simple systems are easier to understand.

Simple systems are easier to repair.

Simple systems are easier to extend.

When in doubt, make the architecture simpler before making the code cleverer.

---

Many components may read published information.

---

Learning

PineRE encourages curiosity.

Users should be able to learn.

Developers should be able to learn.

Contributors should be able to learn.

Knowledge should be discoverable.

Knowledge should be understandable.

Knowledge should not depend upon memory alone.

---

Recovery

Recovery exists to protect the user.

It should repair before replacing whenever practical.

It should preserve user content whenever practical.

Recovery is not an afterthought.

It is a core part of the PineRE experience.

---

Open Development

PineRE learns from others.

We reference.

We understand.

We reimplement.

We improve.

Whenever practical, PineRE prefers small, well-understood changes to upstream projects rather than maintaining large, long-lived forks.

---

The User Experience

Gaming comes first.

Desktop, development and advanced features remain available without overwhelming users who simply wish to enjoy their handheld.

Every feature should improve the experience.

Not complicate it.

---

The Journey

PineRE was not designed from a complete blueprint.

It emerged through careful exploration.

We did not draw the map first and force PineRE to fit it.

We explored the landscape.

The map emerged from the discoveries.

PineRE does not wander.

PineRE surveys.

Every investigation adds another piece to the map.

Every discovery makes the map more accurate.

---

Our Commitment

PineRE aims to remain approachable.

Approachable to users.

Approachable to developers.

Approachable to contributors.

Software should not only work well.

It should also be understandable.

That is the PineRE way.
